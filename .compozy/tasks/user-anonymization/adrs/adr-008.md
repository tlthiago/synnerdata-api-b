# ADR-008: Extend `AuditService.log` with optional transaction parameter

## Status

Accepted

## Date

2026-04-27

## Context

`src/modules/audit/audit.service.ts:9-29` defines:

```typescript
static async log(entry: AuditLogEntry): Promise<void> {
  try {
    await db.insert(schema.auditLogs).values({...});
  } catch (error) {
    logger.error({ type: "audit:log:failed", entry, error });
  }
}
```

The signature accepts no transaction, and the `try/catch` silences insert failures by design — audit logging on regular CRUD is fire-and-forget so a flaky audit pipeline cannot break business operations.

ADR-002 requires the anonymization audit-log insert to participate in the same transaction as the PII overwrite and credential deletions. If the audit insert fails, the entire transaction must roll back. Two contracts conflict: silent-fail (current) vs strict-fail-inside-transaction (anonymization).

Two approaches were considered:

- Extend `AuditService.log` with an optional transaction parameter. When provided, the insert runs on the transaction connection and errors propagate. When absent, current silent-fail behavior is preserved.
- Inline `tx.insert(schema.auditLogs).values({...})` directly in `AnonymizeService.anonymize`, bypassing `AuditService.log` for this one path.

## Decision

Extend `AuditService.log` to accept an optional `tx` parameter:

```typescript
static async log(
  entry: AuditLogEntry,
  tx?: NodePgDatabase<typeof schema> | PgTransaction<...>
): Promise<void>
```

Behavior:

- When `tx` is **provided**: the insert runs on `tx` and **errors propagate** (no `try/catch`). This makes the insert participate in the caller's transaction; a failure rolls back the whole transaction.
- When `tx` is **absent**: current behavior is preserved exactly — insert runs on `db`, errors are caught and logged.

The decision condenses to "if you pass a transaction, you opt into strict semantics; otherwise, fire-and-forget." No separate `strict` flag is added.

The `AnonymizeService` calls `AuditService.log(entry, tx)` from inside `db.transaction(async (tx) => { ... })`.

## Alternatives Considered

### Alternative 1: Inline `tx.insert(schema.auditLogs).values(...)` directly

- **Description**: Skip `AuditService.log` for the anonymization path; write the row by hand inside the transaction body.
- **Pros**: Zero changes to a shared service; smallest possible blast radius.
- **Cons**: Duplicates the field-mapping logic (id generation `audit-${crypto.randomUUID()}`, null-default handling for `organizationId`/`resourceId`/`changes`/`ipAddress`/`userAgent`). Two places to update if the `audit_logs` schema evolves. Future transactional audit callers (e.g., a follow-up PRD on admin-anonymize) would each duplicate or eventually drive a re-extraction.
- **Why rejected**: The duplication is non-trivial (~10 lines of structured field mapping) and the abstraction already exists; a single-arg extension is a smaller change than maintaining two write paths.

### Alternative 2: Add a separate `AuditService.logStrict(entry, tx)` method

- **Description**: Two named methods with different semantics rather than one method with conditional behavior.
- **Pros**: API call site reads more explicitly: "I want strict semantics, on this transaction."
- **Cons**: Two methods to maintain; the only difference is the `try/catch` and the connection. Conditional behavior on a single optional parameter is idiomatic and cleaner.
- **Why rejected**: API surface bloat without semantic gain.

## Consequences

### Positive

- Single audit-write entry point preserved across the codebase.
- Existing callers continue to work without modification — new parameter is optional.
- Future transactional audit needs (admin-anonymize PRD, system-initiated anonymization, anything else that needs guaranteed audit-on-success) reuse the same API.
- The behavioral difference is encoded in the call signature, not in a runtime flag.

### Negative

- The conditional `try/catch` inside `AuditService.log` requires careful documentation so future readers understand why two behaviors live in one method. Mitigation: a 2-line comment at the top of the method explaining the contract.
- An anonymization caller who *forgets* to pass `tx` would silently fall back to fire-and-forget — defeating ADR-002. Mitigation: the call site is `AuditService.log(entry, tx)` inside the transaction body where `tx` is in scope; missing it would be flagged in code review.

### Risks

- A future change that adds caching or batching to `AuditService.log` must respect the strict contract when `tx` is provided. Mitigation: any such change is discussed in a dedicated ADR.

## Implementation Notes

The implementation is a minimal refactor:

```typescript
type DbConn = typeof db | Parameters<Parameters<typeof db.transaction>[0]>[0];

static async log(entry: AuditLogEntry, tx?: DbConn): Promise<void> {
  const conn = tx ?? db;
  if (tx) {
    await conn.insert(schema.auditLogs).values(buildRow(entry));
    return;
  }
  try {
    await conn.insert(schema.auditLogs).values(buildRow(entry));
  } catch (error) {
    logger.error({ type: "audit:log:failed", entry, error });
  }
}
```

`buildRow(entry)` encapsulates the existing field mapping (id generation, null defaults). It is a private helper inside the same file.

A unit test or integration test asserts the strict-mode contract: when `tx` is provided and the insert fails, the error propagates (and the transaction caller can roll back).
