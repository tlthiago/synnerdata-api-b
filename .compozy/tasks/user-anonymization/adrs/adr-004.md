# ADR-004: Module placement under `src/modules/auth/anonymize/`

## Status

Accepted

## Date

2026-04-27

## Context

PRD #2 introduces `POST /v1/account/anonymize`. The endpoint is mounted under the `/v1/account` URL namespace, but its semantics are deeply auth-coupled: it reads/writes Better Auth tables (`users`, `sessions`, `accounts`, `twoFactors`, `apikeys`, `invitations`), reuses the existing `validateUserBeforeDelete` invariant, and replaces a Better Auth-native flow.

Three placement options were considered:

- A new submodule `src/modules/auth/anonymize/` (sibling to `src/modules/auth/__tests__/`).
- A new top-level module `src/modules/account/` aligned with the URL namespace, anticipating future "account" operations (export, change-email, etc.).
- Flat files inside `src/modules/auth/` (`anonymize.service.ts`, `anonymize.controller.ts`).

## Decision

Place the implementation under `src/modules/auth/anonymize/` as a dedicated submodule. The submodule contains:

- `anonymize.controller.ts` — Elysia controller exposing the route.
- `anonymize.service.ts` — orchestrates validation, transactional mutation, and post-commit email send.
- `anonymize.model.ts` — Zod schemas for request/response.
- `__tests__/anonymize.test.ts` — integration coverage.

The route's URL prefix (`/v1/account/anonymize`) is decoupled from the module's filesystem location and is set at controller wiring time (controller `prefix: "/account"` mounted by the app, or full path declared on the route).

## Alternatives Considered

### Alternative 1: New top-level module `src/modules/account/`

- **Description**: Establish a new module bucket aligned with the `/v1/account` URL namespace, anticipating future account-management operations.
- **Pros**: URL-to-filesystem alignment is more discoverable for newcomers; signals intent for future related operations.
- **Cons**: Speculative — no concrete second operation today. Would create a module that imports heavily from `lib/auth/*` and Better Auth tables, blurring the boundary between auth and account. YAGNI.
- **Why rejected**: Premature module fragmentation. When and if a second account operation appears, promoting the submodule to a top-level module is a low-cost refactor.

### Alternative 2: Flat files in `src/modules/auth/`

- **Description**: Drop `anonymize.service.ts` and `anonymize.controller.ts` directly under `src/modules/auth/`.
- **Pros**: Minimum directory depth.
- **Cons**: Pollutes the auth module root with operation-specific files. The auth module already contains shared concerns (sessions, plugins, hooks); mixing operation-specific code reduces local cohesion.
- **Why rejected**: Submodule grouping yields clearer ownership without cost.

## Consequences

### Positive

- Localized blast radius: all anonymization-specific code lives under one directory.
- Tests live next to the implementation following project convention (`__tests__/`).
- Clear extraction path if `account` operations grow: promote the submodule to a top-level module by moving the directory and updating imports.

### Negative

- The route's URL path (`/v1/account/anonymize`) is not directly inferable from the filesystem location. Mitigation: the controller's `prefix` declaration documents the binding explicitly.

### Risks

- None material.

## Implementation Notes

- App wiring registers the controller under the `/v1/account` path prefix.
- `__tests__/anonymize.test.ts` replaces `src/modules/auth/__tests__/delete-account.test.ts` as the canonical deletion test file. The old file is removed in the same PR.
