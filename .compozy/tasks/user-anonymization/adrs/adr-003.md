# ADR-003: Direct rollout (homolog → prod) over feature-flag coexistence

## Status

Accepted

## Date

2026-04-27

## Context

The anonymization endpoint replaces a Better Auth-native flow currently consumed by the frontend (`authClient.deleteUser({ password })`). The design doc rates the change as medium risk because it touches an auth-critical path and requires DPO/jurídico sign-off and frontend coordination.

Three rollout strategies were considered:

- Direct: deploy backend → validate in homolog → deploy prod; frontend swaps the client call in the same cycle.
- Feature flag per environment: new endpoint coexists with `user.deleteUser.enabled` for a window; flag flips per environment with rollback by toggle.
- Coexistence with pilot: new endpoint in prod but frontend keeps calling `deleteUser` until an internal pilot user validates end-to-end.

## Decision

Adopt **direct rollout**. Backend disables `user.deleteUser.enabled` and ships the new endpoint in one deploy cycle. Frontend updates the SDK call to the new endpoint in coordination during the same cycle. Validation gate is manual end-to-end testing in homolog plus DPO/jurídico sign-off before prod deploy.

## Alternatives Considered

### Alternative 1: Feature flag per environment

- **Description**: Keep `user.deleteUser.enabled: true` and add the new endpoint behind a flag. Flip the flag (and disable Better Auth's flow) per environment after validation.
- **Pros**: Granular rollback by toggle; backend can ship without frontend in lock-step.
- **Cons**: Two active deletion paths during the coexistence window; double the support surface; requires explicit cleanup PRD to remove the flag and the deprecated path. Risk of the cleanup never happening.
- **Why rejected**: The change is well-bounded (single endpoint, single frontend call site); the operational cost of dual paths exceeds the rollback flexibility gain.

### Alternative 2: Coexistence with internal pilot

- **Description**: Backend in prod; frontend continues calling `deleteUser` until an internal pilot user validates the new flow end-to-end.
- **Pros**: Most conservative; surfaces issues with a known-tolerant user before customer exposure.
- **Cons**: Stretches the cycle; introduces ambiguity about which path is canonical; pilot signal is weak for an action that happens once per account lifetime.
- **Why rejected**: For an irreversible single-shot action, a pilot of one or two internal accounts is unlikely to surface meaningfully different failure modes than the homolog test matrix; the coordination overhead is not justified.

## Consequences

### Positive

- Single canonical path post-deploy; no flag debt to clean up.
- DPO/jurídico review focuses on one final flow.
- Frontend and backend ship the contract change atomically in the same cycle.

### Negative

- No quick toggle if a critical issue is discovered in prod — rollback requires reverting the deploy (re-enabling `deleteUser` + reverting frontend). Mitigation: comprehensive homolog validation including DPO sign-off; deploy during low-traffic window; on-call awareness during first 24h.
- Frontend and backend cycles must coordinate calendar; if frontend slips, backend cannot ship independently. Mitigation: backend keeps `deleteUser` enabled until frontend is ready; the disable lands in the final deploy.

### Risks

- A bug discovered post-deploy that blocks anonymization for users would block a (rare) operation. Mitigation: 5xx errors surface clearly to the user, who can retry; if the bug is systemic, rollback is the response.

## Implementation Notes

Sequence:

1. Backend deploys to homolog with new endpoint live AND `user.deleteUser.enabled: true` (parallel for testing).
2. Frontend deploys to homolog calling the new endpoint.
3. Manual end-to-end validation against the 9 scenarios from `delete-account.test.ts`.
4. DPO/jurídico sign-off.
5. Backend deploys to prod with `user.deleteUser.enabled: false` and the new endpoint.
6. Frontend deploys to prod calling the new endpoint.
7. Post-deploy SQL invariant check (zero orphans across `sessions`/`accounts`/`twoFactors`/`apikeys`/`invitations` for `users.anonymized_at IS NOT NULL`).
