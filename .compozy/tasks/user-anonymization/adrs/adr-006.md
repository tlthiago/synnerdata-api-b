# ADR-006: Minimal non-PII payload for the anonymization audit-log entry

## Status

Accepted

## Date

2026-04-27

## Context

Each successful anonymization inserts one row into `audit_logs` with `action="anonymize", resource="user", resourceId=<userId>, userId=<self>`. The `changes` JSON column may carry additional context. The PRD's posture is "no original PII retained anywhere after the transaction commits" — the audit row included.

Three payload shapes were considered:

- Minimal non-PII metadata capturing business context (was-owner-of-trial-org, cascade target).
- Empty changes (`null`); the row's existence and identifiers are the entire signal.
- Operational metadata including timestamps and role.

## Decision

The anonymization audit-log row carries:

```jsonc
{
  "before": {
    "wasOwnerOfTrialOrg": <boolean>,
    "organizationCascade": <organizationId | null>
  },
  "after": null
}
```

No `name`, `email`, `image`, or any other PII appears anywhere in the row. The two captured booleans reflect business context that supports forensic review without identifying the person.

## Alternatives Considered

### Alternative 1: Empty changes (`changes: null`)

- **Description**: Insert only `action`, `resource`, `resourceId`, `userId`. The event is its own signal.
- **Pros**: Maximum simplicity; nothing to maintain in the payload schema.
- **Cons**: Loses the cascade signal (was an org also deleted? if so, which?). Forensic reviews would have to cross-join with the `organizations` table — assuming the org row still exists, which it does not after cascade.
- **Why rejected**: The cascade target (organization ID) is non-PII operational signal that disappears from the rest of the database. Capturing it once in the audit row is cheap insurance for forensics.

### Alternative 2: Operational metadata including timestamps and role

- **Description**: `{ before: { role, wasOwnerOfTrialOrg, organizationCascade, accountCreatedAt }, after: null }`.
- **Pros**: Richer forensic picture.
- **Cons**: `accountCreatedAt` already lives on the (preserved) anonymized `users` row — duplication. `role` is debatable: admins are blocked from anonymization, so `role` is always `user` (or other non-admin) at insertion time; capturing it adds no signal.
- **Why rejected**: Each extra field needs a justification; none survive the test.

## Consequences

### Positive

- The audit row is fully LGPD-safe: zero PII, zero linkable identifiers beyond the user ID itself (which references an anonymized row).
- Cascade context is preserved exactly where it is needed: a single row.
- Schema is small and stable; future readers can add fields without breaking existing rows.

### Negative

- The row tells you *that* an anonymization happened and whether an org was cascaded; it does not tell you *why* the user requested it (no `reason` field). Mitigation: `reason` is not collected by the endpoint; if a future PRD adds an optional reason input, the audit payload extends additively.

### Risks

- A future change to `wasOwnerOfTrialOrg` or `organizationCascade` semantics would require care to avoid breaking historical rows. Mitigation: stored as a JSON object with named fields, additions are non-breaking.

## Implementation Notes

- The audit-log insert runs **inside** the same transaction as the anonymization mutations (per ADR-002). This makes the audit row part of the all-or-nothing commit.
- The `AuditService.log()` call is fire-and-forget on top-level errors, but inside a transaction, an insert failure causes the transaction to roll back — which is the desired atomicity.
- The audit-log entry uses `buildAuditEntry()` from `src/lib/auth/audit-helpers.ts` for shape consistency with the rest of the audit pipeline.
