# ADR-009: Two-PR rollout sequencing — coexistence first, then disable

## Status

Accepted

## Date

2026-04-27

## Context

PRD #2 risk section and ADR-003 prescribe: "backend keeps `user.deleteUser.enabled: true` until the frontend is ready; the disable lands in the final deploy. Cycles are coordinated calendar-explicitly." This wording mandates a deploy sequence in which the new endpoint is live for the frontend to consume *before* the legacy Better Auth path is removed.

ADR-005 originally framed the validator refactor and the `user.deleteUser` block removal as same-PR work. With the deploy-sequencing requirement in mind, that framing collapses two distinct events into one and risks a window in production where the frontend is still calling the legacy `deleteUser` endpoint that no longer exists.

Two sequencing strategies were considered:

- Single PR, single deploy: ship everything (new endpoint + disable + frontend swap) in one calendar-coordinated release.
- Two PRs, two deploys: PR 1 adds the new endpoint and refactors the validator while keeping the legacy `deleteUser` block functional; the frontend deploys consuming the new endpoint; PR 2 removes the legacy block.

## Decision

Adopt the **two-PR sequencing**.

**PR 1 — coexistence:**

- Adds `anonymized_at` column migration.
- Adds the `AnonymizeController`, `AnonymizeService`, Zod schemas, email template, sender wrapper.
- Adds `BadRequestError` (if not already present in `src/lib/errors/http-errors.ts`).
- Adds `"anonymize"` to the `AuditAction` enum.
- Refactors `validateUserBeforeDelete` to throw `AppError` (per ADR-005).
- Adds a small adapter inside the existing `auth.ts` `beforeDelete` hook that catches `AppError` and re-throws as Better Auth's `APIError`, preserving the legacy `POST /api/auth/delete-user` flow.
- Mounts the new endpoint at `/v1/account/anonymize`.
- Adds the new integration tests and removes the legacy `delete-account.test.ts`.
- Updates `src/modules/auth/CLAUDE.md` to document both flows during the transition (clearly noting the legacy is on borrowed time).
- Extends `AuditService.log` with optional `tx` parameter (per ADR-008).

After PR 1 deploys, both endpoints exist:

- `POST /api/auth/delete-user` — legacy Better Auth flow, still wired, still usable.
- `POST /v1/account/anonymize` — new flow, live and ready.

**Frontend deploys** consuming the new endpoint. Validation in homolog and prod canary.

**PR 2 — cleanup:**

- Removes `user: { deleteUser: { ... } }` block from `src/lib/auth.ts`, including the adapter.
- Deletes `auditUserDelete` from `src/lib/auth/audit-helpers.ts`.
- Updates `src/modules/auth/CLAUDE.md` to remove the legacy section.

After PR 2 deploys, the legacy endpoint returns 404.

## Alternatives Considered

### Alternative 1: Single PR, single deploy

- **Description**: One PR ships everything — new endpoint, disable, frontend SDK swap — and a coordinated single deploy lands them together.
- **Pros**: Smallest cycle count; no transitional state; legacy gone immediately.
- **Cons**: A single coordinated deploy across backend and frontend depends on perfect calendar alignment. Any slip in the frontend deploy window leaves either (a) a backend with the disable but a frontend still calling the legacy endpoint, or (b) holds the entire backend release until the frontend is exactly ready.
- **Why rejected**: The PRD and ADR-003 already prescribe the "until frontend is ready" sequencing. A coordinated single deploy is fragile; adding a coexistence window is cheap insurance.

### Alternative 2: Feature flag in the backend gating the disable

- **Description**: Keep `user.deleteUser.enabled` driven by an environment flag; ship in one PR and flip the flag per environment.
- **Pros**: Single PR; granular flip per environment; fast rollback.
- **Cons**: Reintroduces feature-flag mechanics that ADR-003 explicitly rejected. Adds env-config debt and a cleanup PR.
- **Why rejected**: Inconsistent with ADR-003.

## Consequences

### Positive

- The new endpoint is live in production before the legacy is removed; the frontend's transition is non-blocking.
- Each PR is reviewable on its own terms — PR 1 is feature-add; PR 2 is dead-code removal.
- The coexistence window provides a natural validation period: any divergent behavior between the two flows surfaces while both are observable.
- Rollback of PR 2 is trivial (revert removes the `auth.ts` block but no behavior change since both paths existed).

### Negative

- Two PRs and two deploys instead of one. Mitigation: PR 2 is small (deletions only) and reviewable in minutes.
- During the coexistence window, two write paths to user deletion exist — observers must understand the difference. Mitigation: the legacy path still runs `validateUserBeforeDelete` and `auditUserDelete`; behavior parity is intact.
- The `auth.ts` adapter (translates `AppError` from `validateUserBeforeDelete` back to `APIError` for Better Auth) is short-lived code that lives only between PR 1 and PR 2. Mitigation: it is one try/catch block, removed entirely with the surrounding `user.deleteUser` definition in PR 2.

### Risks

- If PR 2 is delayed indefinitely, the codebase carries a `auth.ts` adapter and dead-flow code longer than expected. Mitigation: the deploy-gate runbook for PR 1 ends with "schedule PR 2 within 1 week of frontend production rollout."
- If a defect in PR 1's new endpoint forces a rollback after frontend has already swapped to the new endpoint, the legacy endpoint is still available as a degraded fallback (the frontend would have to be rolled back too, but the backend is not blocked).

## Implementation Notes

The PR 1 adapter looks roughly like this:

```typescript
// src/lib/auth.ts (PR 1 — temporary)
user: {
  deleteUser: {
    enabled: true,
    async beforeDelete(user, request) {
      let orgIdToDelete: string | null;
      try {
        orgIdToDelete = await validateUserBeforeDelete(user as AuthUser);
      } catch (error) {
        if (error instanceof AppError) {
          throw new APIError("BAD_REQUEST", { code: error.code, message: error.message });
        }
        throw error;
      }
      if (orgIdToDelete) await auth.api.deleteOrganization({...});
    },
    async afterDelete(user) { await auditUserDelete(user); },
  },
}
```

PR 2 removes the entire `user: { deleteUser: { ... } }` block and the adapter goes with it.
