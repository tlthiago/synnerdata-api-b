# ADR-005: Refactor `validateUserBeforeDelete` to AppError and remove Better Auth `user.deleteUser` block

## Status

Accepted

## Date

2026-04-27

## Context

The pre-deletion guard `validateUserBeforeDelete` lives in `src/lib/auth/hooks.ts` (lines 127-176). It currently throws `APIError("BAD_REQUEST", { code, message })` because it is invoked by Better Auth's `beforeDelete` hook in `src/lib/auth.ts`.

The project convention (root `CLAUDE.md`) mandates the `AppError` hierarchy in `src/lib/errors/` for typed errors raised in controllers. Better Auth's `APIError` is incompatible with the project's response-envelope pipeline.

PRD #2 disables `user.deleteUser.enabled`. Once disabled, the `beforeDelete` hook in `src/lib/auth.ts:101-115` is dead code; the `auditUserDelete` companion in `src/lib/auth/audit-helpers.ts:47-60` becomes obsolete because anonymization records its own audit event in-transaction.

Three refactor strategies were considered:

- Refactor `validateUserBeforeDelete` to throw `AppError` and remove the now-dead Better Auth `user.deleteUser` block.
- Wrap calls to the helper in the new service (try/catch → map to `AppError`), keeping the helper compatible with Better Auth.
- Extract the validation logic into a new `AppError`-native validator and leave the legacy helper intact until a later cleanup.

## Decision

Refactor `validateUserBeforeDelete` to throw the project's `AppError` subclasses (specifically `BadRequestError` with stable `code` strings preserved: `ADMIN_ACCOUNT_DELETE_FORBIDDEN`, `ACTIVE_SUBSCRIPTION`, `ORGANIZATION_HAS_MEMBERS`). Across the two-PR rollout (see ADR-009 for sequencing), **remove** the entire `user: { deleteUser: { ... } }` block from `src/lib/auth.ts` and **delete** `auditUserDelete` from `src/lib/auth/audit-helpers.ts`. Keep `buildAuditEntry` (still used by the new anonymization audit-log call).

In **PR 1** (coexistence): the validator is refactored to `AppError`, but the `user.deleteUser` block stays wired with a small adapter that catches `AppError` and re-throws as `APIError` so the legacy `POST /api/auth/delete-user` continues to work for the frontend during the transition window.

In **PR 2** (cleanup, after frontend rollout): the entire `user.deleteUser` block (including the adapter) and `auditUserDelete` are deleted.

The helper remains in `src/lib/auth/hooks.ts` as the single source of truth for "can this user be removed from the platform?" invariants; the new anonymization service imports and calls it directly.

## Alternatives Considered

### Alternative 1: Wrap calls in new service (mapping `APIError` → `AppError`)

- **Description**: Keep `validateUserBeforeDelete` throwing `APIError`; the new service catches and re-throws as `AppError`.
- **Pros**: No change to the legacy helper; smallest diff in `lib/auth/`.
- **Cons**: Two error hierarchies coexist in the call chain; mapping logic must enumerate every code; project convention is violated by the helper itself; future readers see `APIError` used outside Better Auth contexts.
- **Why rejected**: Disabling Better Auth's `deleteUser` makes the helper's only caller the new endpoint — there is no reason to preserve `APIError` semantics. Dragging the legacy throws forward is debt without benefit.

### Alternative 2: Extract logic into a new validator, leave legacy helper intact

- **Description**: Introduce `src/modules/auth/anonymize/anonymize.validators.ts` with `AppError`-native logic; mark the old helper deprecated.
- **Pros**: Zero diff in `lib/auth/` initially; explicit "new code path" boundary.
- **Cons**: Duplicated validation logic during the deprecation window; risk of drift between the two implementations; cleanup PR required later.
- **Why rejected**: Disabling Better Auth's `deleteUser` makes the legacy helper an orphan immediately. Duplicating to defer cleanup that has no real cost is a YAGNI anti-pattern.

## Consequences

### Positive

- Single error hierarchy (`AppError`) across the project's controllers and services, consistent with `CLAUDE.md`.
- `lib/auth/hooks.ts` becomes the canonical place for the "can-be-removed" invariant, owned by the project rather than Better Auth's hook lifecycle.
- Net code reduction: the dead `user.deleteUser` block, `auditUserDelete`, and any `APIError` import in `hooks.ts` are removed.

### Negative

- Anyone with cached knowledge of the previous `APIError` codes must adjust. Mitigation: error codes (`ADMIN_ACCOUNT_DELETE_FORBIDDEN`, `ACTIVE_SUBSCRIPTION`, `ORGANIZATION_HAS_MEMBERS`) are preserved verbatim; only the error class changes.
- The Better Auth deletion path is gone; if any internal tooling relied on `POST /api/auth/delete-user`, it breaks. Mitigation: search for references at PR time; expected to be zero outside the test that is being rewritten.

### Risks

- A hidden caller of the legacy `auditUserDelete` could break silently. Mitigation: full grep for `auditUserDelete` across `src/`; pre-merge confirmation that the only callsite is `auth.ts:113`.

## Implementation Notes

- Error class mapping: `APIError("BAD_REQUEST", { code, message })` → `BadRequestError(message, { code })` (or a small subclass per code if `BadRequestError` does not carry a custom code field; the existing `AppError` base supports a `code` property).
- The project does not currently expose `BadRequestError`; if missing, add it to `src/lib/errors/http-errors.ts` following the `ConflictError` / `ForbiddenError` pattern (status 400, code `BAD_REQUEST` by default, overridable per instance).
