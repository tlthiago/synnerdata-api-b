# ADR-001: Self-service-only anonymization scope for PRD #2

## Status

Accepted

## Date

2026-04-27

## Context

PRD #2 of the "user attribution on domain resources" roadmap replaces Better Auth's hard-delete user flow with a project-owned anonymization endpoint. The design doc (`docs/improvements/2026-04-27-user-attribution-roadmap-design.md`) binds the technical posture (anonymization over hard delete, query style, schema conventions) but leaves the product scope open.

Three product approaches were considered for the PRD:

- A — MVP enxuto: self-service only (user-initiated anonymization).
- B — Self + admin "anonymize on behalf" (support/DPO-facing endpoint).
- C — Self + framework for system-initiated anonymization (e.g., long-inactive accounts).

The roadmap design doc explicitly defers grace period and adjacent retention concerns to future PRDs (line 51, line 212). Admin-initiated and system-initiated anonymization were not validated by the design and have no concrete demand.

## Decision

Adopt **Approach A**. PRD #2 ships a single self-service endpoint that replaces `authClient.deleteUser({ password })`. No admin-initiated path, no system-initiated framework, no grace period.

## Alternatives Considered

### Alternative 1: Approach B — Self + admin "anonymize on behalf"

- **Description**: Adds `POST /v1/admin/users/:id/anonymize` for support staff or DPO operators executing requests received outside the self-service flow.
- **Pros**: Covers off-channel LGPD requests in the same PRD; avoids a follow-up cycle when the case appears.
- **Cons**: Doubles the API surface and test matrix; introduces RBAC coordination with the admin module; stretches the cycle by ~1 week without validated demand.
- **Why rejected**: No documented demand today; speculative scope expansion violates YAGNI. If demand materializes, a focused follow-up PRD is cheaper than carrying premature surface here.

### Alternative 2: Approach C — Self + framework for system-initiated anonymization

- **Description**: Implements a reusable anonymization service the system can invoke for future automatic policies (inactivity, retention triggers).
- **Pros**: Accelerates downstream retention PRDs.
- **Cons**: Pure YAGNI — design doc explicitly defers retention concerns; no validated trigger today; framework abstraction risks being wrong without a real second consumer.
- **Why rejected**: Violates the roadmap's explicit "future PRD" partitioning. Frameworks built without two concrete consumers tend to be wrong.

## Consequences

### Positive

- Smallest surface that satisfies the roadmap's PRD #2 goal.
- Cycle time aligned with the design doc estimate (~1-2 weeks).
- Test surface mirrors the existing `delete-account.test.ts` matrix, simplifying parity validation.
- DPO/jurídico review covers a single user journey, not a matrix of operator paths.

### Negative

- Off-channel LGPD requests (e.g., user emails legal asking for deletion) require manual workaround until a future admin-anonymize PRD ships. Mitigation: support team can execute via direct DB script if necessary, with audit-log entry recorded manually.
- A future admin-initiated PRD will need to revisit RBAC and reuse decisions taken here, accepting some rework.

### Risks

- If support volume of off-channel requests is non-trivial, the manual workaround becomes a sustained operational cost. Mitigation: track support volume; if non-trivial, prioritize the admin PRD.

## Implementation Notes

The endpoint contract, validation rules, and atomicity semantics are documented in the PRD itself and in ADR-002 (atomicity) and ADR-003 (rollout strategy).
