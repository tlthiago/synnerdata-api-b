# ADR-002: Atomic single-transaction semantics for anonymization

## Status

Accepted

## Date

2026-04-27

## Context

Anonymization performs several mutations against the database:

- Replace PII fields on the `users` row (`name`, `email`, `image`, `email_verified`, `anonymized_at`).
- Delete dependent rows in `sessions`, `accounts`, `twoFactors`, `apikeys`, `invitations`.
- Optionally delete the user's organization if they are the sole owner of an empty trial org (cascade rule preserved from current `beforeDelete` flow).
- Insert one `audit_logs` row capturing the anonymization event.
- Send a confirmation email to the original address (out-of-band, see ADR/PRD).

The PRD must define what happens if any step fails mid-flow.

## Decision

Run all database mutations (PII replacement, dependent-row deletions, optional org cascade, audit-log insert) inside a **single PostgreSQL transaction**. If any step fails, the entire transaction rolls back: the user account remains intact and no `audit_logs` entry is recorded. The endpoint returns a 5xx error and the user can retry.

The confirmation email is sent **after** the transaction commits successfully (best-effort post-commit). Email delivery failure does not roll back the anonymization.

## Alternatives Considered

### Alternative 1: Best-effort with checkpoint and reconciliation endpoint

- **Description**: Replace PII first (irreversible), then delete credentials. If credential deletion fails, mark the operation as "partial" in `audit_logs` and expose an admin endpoint to reconcile. UI shows success once PII is removed.
- **Pros**: User-perceived success even on partial failure; simpler error handling for the synchronous path.
- **Cons**: Anonymized user could retain valid sessions/credentials briefly, allowing login as an "anonymized" identity — a security and UX hazard. Introduces a partial-state machine that must be documented, monitored, and reconciled. Recovery path requires admin tooling not yet planned.
- **Why rejected**: The window where credentials outlive PII is unacceptable for an auth-critical flow. Operational cost of monitoring/reconciliation outweighs the resilience gain.

### Alternative 2: Atomic transaction with automatic retry

- **Description**: Same atomicity as the chosen decision, plus 1-2 automatic retries on transient failures (deadlocks, transient connection loss).
- **Pros**: Smooths over flaky transient errors invisible to the user.
- **Cons**: Anonymization is destructive — silent retries on ambiguous failures could mask legitimate issues; the operation is rare (once per account lifetime), so retry value is low. Retry policy adds complexity.
- **Why rejected**: For a low-volume, high-stakes operation, surfacing the error to the user (who can retry manually) is preferable to silent retry. Can be revisited if telemetry shows transient-failure rate is non-trivial.

## Consequences

### Positive

- No window where an "anonymized" user retains login credentials.
- Single source of truth: `audit_logs` row exists if and only if anonymization succeeded.
- Failure semantics are simple to test (rollback verified by the original row remaining intact).
- Aligns with the design doc's "single DB transaction" intent (line 96).

### Negative

- A transient infra failure (e.g., DB deadlock during cascade) forces the user to retry manually. Acceptable given the low frequency of the operation.
- Confirmation email after commit means a successful anonymization may not be confirmed by email if mail delivery fails. Mitigation: log mail-delivery failures separately; the audit-log entry is still authoritative.

### Risks

- Long-running cascade (user owns org with many cascade-deletable rows) could hit transaction-time limits on very large orgs. Mitigation: validation already blocks owners with members, and trial-org cascade is bounded by business rules; the realistic dataset for cascade is small.

## Implementation Notes

Transaction boundary uses the project's existing Drizzle transaction API (`db.transaction(async (tx) => { ... })`). Email send is enqueued or invoked after `tx` commits; failure of the email pipeline does not affect transaction outcome.
