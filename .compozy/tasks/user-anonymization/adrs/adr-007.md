# ADR-007: Password verification via Better Auth `auth.api.verifyPassword`

## Status

Accepted

## Date

2026-04-27

## Context

The anonymization endpoint requires the user to re-enter their password (PRD User Story; security default for irreversible operations). With Better Auth's native `deleteUser` flow disabled (ADR-005), the project must verify the password against the credentials account itself.

Three approaches were considered:

- Use Better Auth's official `auth.api.verifyPassword({ body: { password }, headers })`.
- Query the `accounts` table directly and verify the hash with `Bun.password.verify`.
- A non-persistent `auth.api.signInEmail` call (verify by side effect, discard the resulting session).

**API contract (verified directly against installed Better Auth 1.6.9 source at `node_modules/better-auth/dist/api/routes/password.mjs:167-193`):**

- The endpoint is `POST /verify-password`. Request body: `{ password: string }`. Headers must carry the active session cookie.
- **Success path**: Better Auth's `validatePassword` returns truthy → handler returns `ctx.json({ status: true })`. The response shape is `{ status: true }` — **not** `{ isValid: true }` as published in some doc snippets.
- **Failure path**: `validatePassword` returns falsy → handler **throws** `APIError("BAD_REQUEST", BASE_ERROR_CODES.INVALID_PASSWORD)`. The caller never sees a `{ status: false }` or `{ isValid: false }` body.
- **Middleware**: `use: [sensitiveSessionMiddleware]` (defined at `node_modules/better-auth/dist/api/routes/session.mjs:307-314`). This middleware re-fetches the session from the database with `disableCookieCache: true` to ensure a session revoked elsewhere cannot pass via cached cookies. It does **not** require a freshly-authenticated session (that is `freshSessionMiddleware` at line 334+, a different middleware). The endpoint's UX flow (modal → password input) does not need to trigger any re-login or "fresh session" handshake — a normal logged-in user with a valid (un-revoked) session passes.

## Decision

Use `auth.api.verifyPassword({ body: { password }, headers: requestHeaders })` inside the anonymization service, before entering the database transaction. Wrap the call in `try/catch`:

- On success the call resolves; the service proceeds.
- On wrong password the call **throws** `APIError("BAD_REQUEST", "INVALID_PASSWORD")`. The service catches it, identifies the `INVALID_PASSWORD` code, and re-throws as `BadRequestError` with the same code. Other `APIError`s and unexpected errors propagate as 5xx.

The service **does not** read the response body or check for `isValid`. The boolean `{ status: true }` is informational; the absence of a thrown error is the success signal.

## Alternatives Considered

### Alternative 1: Direct query of `accounts.password` + `Bun.password.verify`

- **Description**: `SELECT password FROM accounts WHERE userId=? AND providerId='credential'`; compare with `Bun.password.verify(input, hash)`.
- **Pros**: No dependency on Better Auth's internal API surface.
- **Cons**: Duplicates knowledge of Better Auth's hashing algorithm and parameters; if Better Auth changes the algorithm or upgrades the hash format, this code silently breaks. Re-implements rate-limiting / breach-detection logic that Better Auth may add in future versions.
- **Why rejected**: Reinventing security primitives is exactly what Better Auth exists to avoid.

### Alternative 2: Non-persistent `auth.api.signInEmail`

- **Description**: Call `auth.api.signInEmail({ body: { email, password }, asResponse: true })`, ignore the resulting session token.
- **Pros**: Reuses an extensively battle-tested code path.
- **Cons**: Side-effectful (creates and orphans a session row); subject to sign-in rate-limiting policies that may differ from password-verify policies; semantically wrong (we are not signing in, we are verifying).
- **Why rejected**: A purpose-built helper exists; using a side-effectful workaround is worse on every axis.

## Consequences

### Positive

- Single, official entry point for password verification; the project's anonymization flow inherits any future hardening Better Auth adds.
- Server-only invocation matches Better Auth's security posture.
- The `sensitiveSessionMiddleware` automatically protects against revoked-but-cached sessions sneaking through — a defense in depth the project gets for free.

### Negative

- The success/failure semantics are throw-vs-resolve, not boolean return — slightly less ergonomic than a `boolean` API but matches Better Auth's idioms.
- A version bump of Better Auth that changes the API signature would require a small adjustment. Mitigation: the call site is one line, easy to update.

### Risks

- If `verifyPassword` itself is rate-limited globally, an attacker repeatedly probing the endpoint could trigger a denial of password-verify for legitimate users. Mitigation: the anonymize endpoint is itself authenticated (requires a valid session), so probing requires a stolen session. Session-loss is a separate threat model.
- If a future Better Auth release changes the error shape (e.g., swaps `BASE_ERROR_CODES.INVALID_PASSWORD` for a new identifier), the catch-block matching must be updated. Mitigation: the catch matches by code; a version-bump checklist includes verifying the code constant.

## Implementation Notes

```typescript
import { APIError } from "better-auth/api";

async function verifyPasswordOrThrow(password: string, headers: Headers): Promise<void> {
  try {
    await auth.api.verifyPassword({ body: { password }, headers });
  } catch (error) {
    if (error instanceof APIError && error.body?.code === "INVALID_PASSWORD") {
      throw new BadRequestError("Senha incorreta.", { code: "INVALID_PASSWORD" });
    }
    throw error;
  }
}
```

The `headers` parameter receives the Elysia request's headers (carrying the session cookie). Pattern at the call site: `await verifyPasswordOrThrow(input.password, input.requestHeaders)`.

The installed Better Auth version 1.6.9 (verified via `node_modules/better-auth/package.json`) exposes the `verifyPassword` endpoint as documented in this ADR. Confirm at implementation time that no version drift has occurred.
