# ADR-003: Service Query Pattern — Drizzle Relational API (`db.query` + `with`)

## Status

Accepted

## Date

2026-04-22

## Context

ADR-001 (delivery approach) and ADR-002 (API contract shape) leave open the question of how services should query the database to populate `createdBy`, `updatedBy`, and `deletedBy` as `{ id, name }` objects.

The project currently uses Drizzle's core query builder exclusively (`db.select()` + `db.insert()` + `db.update()` + `.returning()`). `db.query.X.findFirst({ with: ... })` is not used in any module — codebase exploration found zero occurrences of the Relational Query API.

The `db` instance is already configured with `fullSchema` (see `src/db/index.ts:10-13`), so the Relational API is available immediately with no setup change. `relations()` definitions will be added in every domain schema file as part of Phase 1; without `db.query`, those relations would exist only to satisfy the FK declarations and would be otherwise unused.

Each domain service needs to self-join `users` three times (for `createdBy`, `updatedBy`, `deletedBy`). With the core query builder, this requires three `aliasedTable(users, "...")` declarations and three `leftJoin` clauses per query, plus manual shaping in the `.select({ ... })` projection.

## Decision

Adopt Drizzle's **Relational Query API** (`db.query.X.findFirst({ with })` and `db.query.X.findMany({ with })`) as the canonical pattern for any read that populates nested user attribution on domain resources.

Relation naming inside schema files is `createdByUser`, `updatedByUser`, `deletedByUser` (per ADR-002 implementation notes). Column selection is declarative via `with: { createdByUser: { columns: { id: true, name: true } } }`.

Write paths (INSERT/UPDATE/DELETE) follow a **write-then-reread** pattern: execute the mutation with `.returning({ id: ... })` inside a transaction, then call `db.query.X.findFirst({ with })` to load the resource with user objects for the response. The second query is a PK lookup on an already-indexed column with two indexed JOINs — expected single-digit millisecond overhead.

The core query builder (`db.select` + `leftJoin`) remains valid for existing services that do not need user attribution and for future services that genuinely need capabilities the Relational API does not support (complex aggregates, window functions, unusual projections).

## Alternatives Considered

### Alternative 1: Manual `db.select()` + `aliasedTable()` + `leftJoin()`

- **Description**: keep every service using the existing core query builder pattern. Add three `aliasedTable(users, "...")` aliases per service to disambiguate the three self-joins with users. Shape the nested object manually inside `.select({ createdBy: { id: aliasA.id, name: aliasA.name }, ... })`.
- **Pros**: consistent with the 100 percent of existing services; SQL is explicit and inspectable; no new API to learn.
- **Cons**: ~20 additional lines of boilerplate per module (aliases + joins + projection); `relations()` definitions become infra without consumers; Drizzle-recommended idiom ignored; error-prone manual column mapping multiplied across 20+ modules.
- **Why rejected**: the boilerplate cost multiplies across every module touched in Phase 3. Since Phase 1 is adding `relations()` anyway, using them is free; ignoring them wastes work already committed.

### Alternative 2: Raw SQL via Drizzle's `sql` template

- **Description**: write CTE-based SQL for each service using `sql` tagged template.
- **Pros**: full control over query shape.
- **Cons**: loses type safety on the nested shape; every service becomes a handwritten SQL string; worst of both worlds compared to the two first-class APIs.
- **Why rejected**: not justified by any specific need; no service has requirements that the structured APIs cannot meet.

## Consequences

### Positive

- **Less code per module**: `db.query.costCenters.findFirst({ with: {...} })` replaces ~20 lines of aliases + joins + projection with ~8 lines.
- **Declarative shape**: `with: { createdByUser: { columns: { id: true, name: true } } }` expresses intent directly; refactoring the exposed fields is a one-line change.
- **Type inference cascades**: the returned object is automatically typed as `{ id, name, ..., createdByUser: { id, name } | null }` — no manual alignment with Zod response schemas.
- **Leverages `relations()`**: the relation definitions added in Phase 1 serve a real consumer, not just the FK metadata.
- **Optimized SQL generated** (since Drizzle v0.28): the Relational API emits `LEFT JOIN LATERAL` + `json_build_array`, comparable to manual `leftJoin` in performance.

### Negative

- **New pattern to learn**: developers familiar with `db.select` will encounter `db.query` syntax for the first time. Mitigation: the pilot module (cost-centers) serves as copy-paste reference; the API is small (`findFirst`, `findMany`, `where`, `with`, `columns`).
- **Write path uses two queries instead of one**: one mutation + one re-read. Overhead is low (PK lookup + two indexed joins on small `users` table) but measurable versus pure `.returning()`. See ADR-002 for rationale.

### Risks

- **Risk: edge case in Drizzle's Relational API with nullable `with` targets produces unexpected SQL.**
  Mitigation: the pilot (cost-centers) must include integration tests covering `null` creator/updater cases; if behavior differs from expected, surfaces before rollout.
- **Risk: a future service needs something `db.query` cannot express.**
  Mitigation: the core query builder remains available; nothing prevents hybrid use in a single service.

## Implementation Notes

- Canonical read pattern:
  ```ts
  return db.query.costCenters.findFirst({
    where: (t, { eq, and, isNull }) =>
      and(eq(t.id, id), eq(t.organizationId, orgId), isNull(t.deletedAt)),
    with: {
      createdByUser: { columns: { id: true, name: true } },
      updatedByUser: { columns: { id: true, name: true } },
      deletedByUser: { columns: { id: true, name: true } },
    },
  });
  ```
- Canonical write-then-reread pattern:
  ```ts
  const [{ id: newId }] = await db
    .insert(schema.costCenters)
    .values({ ... })
    .returning({ id: schema.costCenters.id });
  return CostCenterService.findByIdWithAudit(newId, organizationId);
  ```
- Deletion of the soft-delete cascade: `deletedByUser` is only non-null on records returned by the delete endpoint or on archived-views. Normal list/detail responses filter `isNull(deletedAt)` and the field returns `null`.
