# ADR-004: Migration Strategy — Single File with `NOT VALID` + `VALIDATE CONSTRAINT`

## Status

Accepted

## Date

2026-04-22

## Context

Phase 1 of the delivery approach (ADR-001) adds foreign keys with `ON DELETE SET NULL` on the audit columns (`created_by`, `updated_by`, `deleted_by`) of 26 domain tables. The production database carries 930 populated audit references as of the 2026-04-21 baseline, all confirmed as pointing to valid users in the `users` table.

Postgres requires a lock on a table to add a foreign-key constraint. The default `ALTER TABLE ... ADD CONSTRAINT ... REFERENCES` acquires `AccessExclusiveLock` for the duration of the constraint creation and validation. On tables with millions of rows this can stall concurrent queries; on the smaller tables in this project it is still a cautious operation in production.

Postgres supports splitting the operation into two phases:

1. `ALTER TABLE ... ADD CONSTRAINT ... NOT VALID` — adds the constraint definition immediately. Takes a brief lock (milliseconds), enforces the constraint for **new and updated rows** going forward, but does not scan existing rows.
2. `ALTER TABLE ... VALIDATE CONSTRAINT` — scans existing rows and marks the constraint as validated. Takes `ShareUpdateExclusiveLock`, which does **not** block reads or writes.

The drizzle-kit tooling generates `ALTER TABLE ... ADD CONSTRAINT ... REFERENCES` by default — the safer `NOT VALID + VALIDATE` variant is not emitted automatically. The 38 existing migrations in `src/db/migrations/` are all auto-generated; zero manually edited.

Given:

- The production audit confirmed **zero orphans** — `VALIDATE` will succeed on first attempt across every table
- Total audit references are small (930 rows distributed across 72 columns × 26 tables) — `VALIDATE` completes in milliseconds per table
- Splitting into two files would require two deployment events, adding operational complexity without measurable benefit at this scale
- Keeping `NOT VALID + VALIDATE` in one file preserves atomicity: a failure aborts the transaction and nothing ships broken

The question is whether to keep the `NOT VALID + VALIDATE` pattern at all (since the tables are small) or emit default FKs, and how to structure the file or files.

## Decision

Generate the migration with `drizzle-kit generate`, then **manually edit the resulting SQL file** to use the `NOT VALID + VALIDATE CONSTRAINT` pattern inline within the same file. Phase the SQL in two blocks:

1. All `ALTER TABLE ... ADD CONSTRAINT ... FOREIGN KEY ... REFERENCES "public"."users"("id") ON DELETE SET NULL NOT VALID;` statements first (for all 26 tables × applicable audit columns)
2. All `ALTER TABLE ... VALIDATE CONSTRAINT ...;` statements next (same order)

Both blocks ship in a single migration file (e.g., `0038_audit_fk_references.sql`), which Drizzle's migrator runs in one transaction. The file is committed to the repository; subsequent drizzle-kit runs must preserve it.

The migration file is the **single authoritative artifact** for the schema change in production.

## Alternatives Considered

### Alternative 1: Two separate migration files (NOT VALID then VALIDATE, deployed separately)

- **Description**: `0038_audit_fk_not_valid.sql` contains only the `ADD CONSTRAINT NOT VALID` statements. `0039_audit_fk_validate.sql` contains only the `VALIDATE CONSTRAINT` statements, deployed in a later window.
- **Pros**: surgical rollback (can drop just the NOT VALID constraints without touching VALIDATE); allows running VALIDATE in a chosen low-traffic window if the validation were slow.
- **Cons**: two deployment events to coordinate; with ~900 audit references total, the VALIDATE step runs in milliseconds and needs no special window; adds operational overhead for no real gain.
- **Why rejected**: the scale of the production data does not justify the split. `VALIDATE CONSTRAINT` on 930 rows across 26 tables is a sub-second operation.

### Alternative 2: Plain FK without `NOT VALID` (let drizzle-kit emit default SQL)

- **Description**: accept the auto-generated `ALTER TABLE ... ADD CONSTRAINT ... REFERENCES ...` without `NOT VALID`. Relies on Postgres acquiring `AccessExclusiveLock` during validation.
- **Pros**: zero manual editing; migration is reproducible from `drizzle-kit generate` alone.
- **Cons**: takes `AccessExclusiveLock` during the full validation scan — acceptable on small tables today, but establishes a pattern that fails as tables grow. Also produces an inconsistent practice: any future migration on a larger table needs the careful pattern anyway, and muscle memory would be wrong.
- **Why rejected**: `NOT VALID + VALIDATE` is the standard Postgres pattern for safe FK addition in production. Establishing it now sets the correct institutional default for future migrations on larger tables.

### Alternative 3: Per-module migration files (one FK per file)

- **Description**: generate 26 migration files, each adding FKs to one table.
- **Pros**: granular rollback at table level.
- **Cons**: 26 files to review, 26 deployment events, fragmented atomicity; the infra change is conceptually one operation.
- **Why rejected**: fragmentation has no benefit given the change is atomic in intent and the data is clean.

## Consequences

### Positive

- **Minimized lock time**: `ADD CONSTRAINT NOT VALID` takes a brief lock per table (each block within milliseconds). `VALIDATE CONSTRAINT` does not block reads or writes.
- **Atomic rollback**: the entire migration is one transaction in Drizzle's migrator. A failure anywhere aborts and leaves the schema unchanged.
- **Zero-downtime deployment**: the schema change ships inside a normal deploy window with no service interruption.
- **Correct institutional default**: future migrations on larger tables reuse the established pattern without having to re-evaluate.

### Negative

- **Manually-edited migration file**: the file generated by drizzle-kit is overwritten with the `NOT VALID + VALIDATE` variant. This is a one-time edit and is committed to the repo like any migration, but it is the first such manual edit in the project's migration history.
- **Risk of drizzle-kit regeneration**: a future `drizzle-kit generate` must not overwrite this migration. Drizzle-kit only emits migrations for new schema diffs, so once the FK exists in the schema files, no future generation will touch `0038`. Verified by convention: drizzle never rewrites completed migrations.

### Risks

- **Risk: `VALIDATE CONSTRAINT` fails during the migration transaction.**
  Mitigation: the pre-migration orphan audit on 2026-04-21 confirmed zero orphans. A failure would indicate a new orphan created between the audit and the migration (exceedingly unlikely); rollback is atomic and the issue can be investigated before retry.
- **Risk: manual edits to the generated migration diverge from the TypeScript schema definition and cause drift.**
  Mitigation: the TypeScript schema uses `.references(() => users.id, { onDelete: "set null" })` — that is the source of truth. The manual `NOT VALID` addition is a deployment-time concern only; the constraint defined is equivalent. Subsequent drizzle-kit generations compare against the TS schema, not the migration body, so the edit is invisible to future diffs.
- **Risk: migration transaction becomes large (many ALTER TABLE statements).**
  Mitigation: no DDL statement holds significant data; the transaction is DDL-heavy but data-light. Postgres handles this class of transaction trivially.

## Implementation Notes

- Generation workflow:
  1. Update TypeScript schemas (26 files) with `.references(() => users.id, { onDelete: "set null" })` and `relations()` additions.
  2. Run `bun db:generate` (invokes `drizzle-kit generate`) to produce the raw migration file.
  3. Manually edit the generated file: change each `ADD CONSTRAINT ... REFERENCES ... ON DELETE SET NULL;` to `... ON DELETE SET NULL NOT VALID;` and append the corresponding `ALTER TABLE ... VALIDATE CONSTRAINT "<constraint_name>";` statements at the bottom.
  4. Verify locally with `bun db:migrate` against a fresh DB; verify against a seeded copy of production via staging.
- Deployment sequence:
  1. Pre-deploy: re-run the orphan audit script (already executed 2026-04-21) to confirm no drift.
  2. Deploy PR 1 (migration + schema changes + test updates).
  3. Post-deploy: run orphan audit again (should still be zero); monitor latency and error rate for 48 hours.
- Rollback: if needed, write a new migration `0039_revert_audit_fk.sql` with `ALTER TABLE ... DROP CONSTRAINT ...` statements. Drizzle-kit does not support automatic down migrations; roll-forward with a reverting migration is the project convention.
