# ADR-002: API Contract Shape for User Attribution Fields

## Status

Accepted

## Date

2026-04-21

## Context

With ADR-001 setting the delivery approach, the project needs to define the public API contract for user attribution on domain resources. Multiple decisions shape the contract:

- Which user fields to expose (`id`, `name`, `email`, `image`, or contextual exposure by role)
- How to name the fields in the payload (semantic `creator`/`updater`/`deleter` vs DB-aligned `createdBy`/`updatedBy`/`deletedBy`)
- When to include `deletedBy` (only on delete/archive endpoints or always present)
- Whether write endpoints (POST/PUT) return the resource already populated with user attribution objects, or a lighter payload

These decisions affect 26 domain tables (all in-scope modules that expose them via CRUD endpoints), the frontend type definitions, and test fixtures across the codebase. Making them upfront avoids contract drift during rollout.

## Decision

The public API contract for user attribution is defined as follows:

1. **Exposed user fields**: `{ id, name }` only.
2. **Field naming**: `createdBy`, `updatedBy`, `deletedBy` â€” same vocabulary as the database columns, now carrying an object instead of a string ID.
3. **Presence**: all three fields are always present in every response that carries a domain resource (`null` when the referenced user is missing or was never set).
4. **Write consistency**: POST and PUT endpoints return the resource with `createdBy`/`updatedBy`/`deletedBy` populated as objects â€” the same shape as GET responses.

### Payload example

```json
{
  "id": "cost-center-abc123",
  "organizationId": "org-xyz",
  "name": "TI",
  "createdBy": { "id": "user-1", "name": "JoĂŁo" },
  "updatedBy": { "id": "user-2", "name": "Maria" },
  "deletedBy": null,
  "createdAt": "2026-04-01T...",
  "updatedAt": "2026-04-15T..."
}
```

## Alternatives Considered

### Alternative 1: Richer user fields (`id + name + email` or `+ image`)

- **Description**: expose email and/or avatar URL in the nested object.
- **Pros**: richer UI options (avatar rendering, disambiguation via email for duplicate names).
- **Cons**: email is PII; exposing it broadly (to every user who can read a resource they did not create) is a privacy regression that requires careful review per module.
- **Why rejected**: `id + name` is sufficient for the primary user need ("who created this"); frontend can fetch richer profile info on demand for specific UIs (e.g., admin screens).

### Alternative 2: Semantic field names (`creator`, `updater`, `deleter`)

- **Description**: rename payload fields to be more idiomatic in English.
- **Pros**: reads naturally in API consumers.
- **Cons**: introduces a second vocabulary disconnected from the DB columns; frontend devs navigating between logs, DB errors, and API responses must translate `creator â†” created_by` mentally.
- **Why rejected**: consistency DBâ†”API is more valuable than linguistic elegance at this scope.

### Alternative 3: `deletedBy` only on delete/archive endpoints

- **Description**: omit `deletedBy` from GET list/by-id responses since those filter `isNull(deletedAt)` and the field would always be null.
- **Pros**: slightly cleaner payload on high-traffic endpoints.
- **Cons**: frontend ends up with two TypeScript types for the same resource (with/without `deletedBy`); type unions proliferate across components.
- **Why rejected**: shape consistency across all endpoints outweighs the tiny payload saving of one always-null field.

### Alternative 4: Differentiated write payload (POST/PUT returns minimal, GET returns full)

- **Description**: write endpoints skip the JOIN with users to save a query; only read endpoints populate objects.
- **Pros**: saves one query per write.
- **Cons**: forces frontend to refetch after create/update to render the card, trading an internal query for an external round-trip; two TypeScript shapes per resource; fragile contract.
- **Why rejected**: the write path is not a hot code path in this product; the consistency gain for frontend and API consumers outweighs one extra query on write.

## Consequences

### Positive

- **Single TypeScript type per resource**: frontend defines one shape, used everywhere.
- **No refetch pattern after writes**: the response of POST/PUT is directly usable to render the updated resource.
- **DBâ†”API vocabulary alignment**: debugging, logs, and API responses share names.
- **Privacy-friendly default**: no PII leaked to users outside of admin flows.

### Negative

- **+1 query on write endpoints**: POST/PUT now need to re-read the resource with `with` clause to populate objects (previously used `.returning()` directly).
- **Mandatory `null` handling in response**: frontend must render `createdBy: null` gracefully (e.g., "System" or blank).

### Risks

- **Risk: performance regression on write-heavy modules.**
  Mitigation: prod has 930 total audit refs; write throughput is low. Baseline benchmarks are recommended on the pilot module before broad rollout. The extra read is a PK + indexed JOIN, expected to add <5 ms per request.
- **Risk: frontend does not handle `null` values and crashes.**
  Mitigation: pilot PR explicitly tests response with `createdBy: null` (system-created records); frontend contract review before pilot merges.

## Implementation Notes

- Service pattern: after `INSERT`/`UPDATE` with `.returning({ id })`, re-fetch with `db.query.<table>.findFirst({ where: eq(..., id), with: { createdByUser: { columns: { id: true, name: true } }, updatedByUser: { columns: { id: true, name: true } }, deletedByUser: { columns: { id: true, name: true } } } })`.
- Response Zod schemas replace `createdBy: z.string().nullable()` with `createdBy: z.object({ id: z.string(), name: z.string() }).nullable()` and mark the field required (present, possibly null).
- Relation naming in Drizzle `relations()`: use `createdByUser`, `updatedByUser`, `deletedByUser` to disambiguate from the payload field names (which are `createdBy`, etc).
