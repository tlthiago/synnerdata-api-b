# ADR-001: Delivery Approach — Infra First + Pilot + Incremental Rollout

## Status

Accepted

## Date

2026-04-21

## Context

The project has 26 domain tables in production with audit columns (`created_by`, `updated_by`, `deleted_by`) stored as plain `text` without foreign key references to `users.id` and without Drizzle `relations()` definitions. Orphan audit in production returned **930 populated references and zero orphans** across all tables, confirming the data is clean and an FK migration is safe.

The work required spans:

1. Schema change in 26 tables (add FK with `onDelete: set null` + `relations()`)
2. One SQL migration (`NOT VALID` + `VALIDATE CONSTRAINT` pattern)
3. New response shape exposing `createdBy`/`updatedBy`/`deletedBy` as `{ id, name }` objects across every domain module
4. Service changes to use `db.query.X.findFirst({ with })` instead of `db.select()`
5. Test updates: factories must insert real users before audit rows; response schemas change for 26 modules
6. Production deployment with customer data

The team must decide how to slice this work into deliverable units (PRs / deploys) balancing risk isolation, review quality, and rollout speed.

## Decision

Deliver the work using the **"Infra First + Pilot + Incremental Rollout"** approach:

- **PR 1 (infra)**: schema-wide change adding FK + `relations()` to all 26 tables, plus the SQL migration. Zero API-visible change. Establishes referential integrity and enables ergonomic `with` queries.
- **PR 2 (pilot feature)**: applies the new response contract in the `cost-centers` module end-to-end — service reads with `relations`, response schema exposes `{ id, name }` objects, tests updated. Serves as canonical reference for remaining modules.
- **PR 3+ (incremental rollout)**: rolls the pilot pattern out to the remaining 26 modules in small PRs (one module or small group per PR), ordered by product demand.

## Alternatives Considered

### Alternative 1: Big Bang (single PR for everything)

- **Description**: one branch/PR covering the 26 tables across schema, migration, services, responses, and tests simultaneously.
- **Pros**: consistency lands atomically; one deploy window covers everything.
- **Cons**: PR touches hundreds of files; review cannot be meaningful; rollback forces reverting good parts along with bad; risk concentrated in a single deploy event.
- **Why rejected**: the migration in production is sensitive and deserves its own focused review; mixing it with 26 response-shape changes dilutes review attention where it matters most.

### Alternative 2: Module by Module (infra + feature per module)

- **Description**: each module becomes a self-contained PR with its own FK addition, migration, and feature change.
- **Pros**: small, self-contained PRs; each delivery brings visible value.
- **Cons**: fragments the DB migration into 26 separate production events instead of one; referential integrity arrives unevenly across the schema; repeated setup overhead (drizzle-kit generate, migration review) for every module.
- **Why rejected**: the `NOT VALID + VALIDATE` migration is the kind of operation you want to do **once, correctly**, not 26 times. Keeping migration atomic is safer than spreading it across months.

## Consequences

### Positive

- **Risk isolation**: PR 1 carries only migration/schema risk (reversible without touching features); PR 2+ carries only feature risk (reversible without touching DB integrity).
- **Pilot validates the pattern**: if the new payload shape or query shape has a flaw, it surfaces in PR 2 (one module) before replicating to 26 others.
- **Review quality**: each PR has focused scope. PR 1 reviewers focus on SQL/migration; PR 2+ reviewers focus on API contract and tests.
- **Granular rollback**: a bug in a rollout PR reverts one module without touching infra.
- **Demand-driven rollout**: modules that have no immediate product need for user attribution do not block the infra change.

### Negative

- **Multiple PRs over time**: more Git overhead than a single delivery.
- **Temporary inconsistency**: during the rollout window, some modules return `{ createdBy: { id, name } }` objects while others still omit the field. Frontend must tolerate both shapes until rollout completes.

### Risks

- **Risk: rollout stalls and a long tail of modules never gets migrated.**
  Mitigation: track rollout status in a checklist; every new feature PR touching a legacy module triggers adoption of the new pattern.
- **Risk: the pilot reveals the pattern needs adjustment after PR 2 is already merged.**
  Mitigation: pilot covers full vertical slice (schema → service → response → tests) so issues surface early; any adjustment is localized to one module before fanout.
- **Risk: frontend team deploys against partially-rolled API and breaks on modules still returning legacy shape.**
  Mitigation: the new fields are **additive** on modules not yet touched — existing payloads keep working; frontend opts into the new fields where they exist.

## Implementation Notes

- PR 1 uses `ALTER TABLE ... ADD CONSTRAINT ... NOT VALID` followed by `ALTER TABLE ... VALIDATE CONSTRAINT` for each FK, minimizing lock contention during deploy. With 930 refs in prod, `VALIDATE` completes in milliseconds per table.
- `relations()` additions are TypeScript-only and do not require a SQL migration.
- Test factories (`UserFactory`, `OrganizationFactory`) already exist and are stable; PR 1 updates tests that insert audit rows with fake user IDs to use real users from factories.
- The pilot module is **cost-centers** because it is the simplest CRUD surface in the repo: one entity, no M2M, no cross-domain joins.
